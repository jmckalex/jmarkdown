File inclusion

JMarkdown can splice the contents of one Markdown file into another. Place the include directive on a line by itself:

[[chapter-one.md]]

Before parsing begins, jmarkdown replaces the directive with the raw content of chapter-one.md. Markdown is inserted before processing, so any syntax in the included file — directives, footnotes, tables, math — is parsed exactly as if it had been written inline at that point in the source.

Path resolution

Relative paths resolve against the directory of the file that contains the directive, not the directory jmarkdown was invoked from. So this layout

book/
  manuscript.md         ← contains [[chapters/intro.md]]
  chapters/
    intro.md            ← contains [[../appendix.md]]
  appendix.md

works whether you run jmarkdown process book/manuscript.md from the project root, from /tmp, or from anywhere else. Each include is resolved relative to its own file: manuscript.md's include points into chapters/, and intro.md's include climbs back out to appendix.md next to the manuscript.

Absolute paths are accepted verbatim. Only .md is recognised; the closing ]] must immediately follow .md.

Syntax rules

  • The directive must occupy a line by itself. Optional leading or trailing whitespace is permitted; anything else on the line (including a wrapping paragraph) leaves the directive literal.
  • Directives inside fenced code blocks or display math ($$ … $$) are not expanded. This page demonstrates the syntax in fenced blocks without any escape trick. (JMarkdown does not recognise 4-space-indent code blocks, so an indented directive on its own line will expand.)
  • Inclusion is recursive: an included file's own [[…]] directives are expanded against that file's directory.

Cycles and re-use

The same file may be included at as many independent sites as you like — the contents are spliced in at each site:

[[boilerplate-header.md]]

# Chapter one
…

[[boilerplate-footer.md]]

What is not allowed is a cycle inside the active expansion chain. If a.md includes b.md and b.md tries to include a.md, jmarkdown prints a warning naming the cycle, leaves the offending [[…]] literal in the output, and continues. Symbolic links are canonicalised, so two paths to the same underlying file still count as the same file for cycle detection.

A safety net of 20 levels of nesting bounds runaway recursion even when cycle detection is somehow defeated.

Errors and missing files

If a referenced file cannot be read, jmarkdown writes an error to standard error naming both the missing path and the including file, and leaves the original [[…]] token in the rendered output as a visible telltale.

What is not supported

  • Metadata headers in included files are not re-parsed. Included content is treated as raw Markdown from the first byte, so do not put a ----fenced YAML block at the top of a file that you intend to include.
  • There is no line-range or partial-file syntax — the whole file is included.
  • There is no conditional include (e.g., "include only when building LaTeX"). Use the LaTeX/HTML conditional directives (:::TeX, :::HTML) inside the included file if you need format-specific content.
