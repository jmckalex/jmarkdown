Live Markdown demos (markdown-demo)

Almost every page in this manual shows a snippet of JMarkdown source beside the output it produces. Those side-by-side panels are not screenshots and they are not maintained by hand. They are generated by a single container directive, markdown-demo, that takes one block of source, displays it and renders it, and lays the two results out next to each other.

The point is a single source of truth. You write the example once; the demo shows the literal source on the left and runs that exact same source to produce the output on the right. The two can never drift apart, because there is only one copy. When you change the example, both panels change together. This is the same discipline JMarkdown applies to a whole book — one source, two outputs — applied in miniature to a documentation snippet.

markdown-demo is a container directive. If the container syntax (the colon-delimited fences) is new to you, read Directives first; this page assumes you know what a container directive is and concentrates on what is special about this one.

A first demo

Place your example between an opening fence naming the directive and a matching closing fence, exactly as you would for any container directive:

:::markdown-demo
:span[This is a coloured span.]{.goldenrod}

1. A list *containing markdown code*
2. Isn't that /cool/?
   a. A sublist item with math $\alpha + \beta$.
   b. Another sublist item.
3. So there you are.
:::

That produces the panel below. The left pane is the source, syntax-highlighted; the right pane is what JMarkdown makes of it:

That contrast is the whole value of the directive: the reader sees both the cause and the effect at once.

What the directive emits

markdown-demo renders to a single grid container with four children — two labels and two panes. Knowing the structure helps if you want to restyle the demos for your own site:

<div class='markdown-demo-container'>
  <div class='markdown-demo-code-label'>Markdown code</div>
  <div class='markdown-demo-output-label'>Markdown output</div>
  <div class='markdown-demo-markdown'>
    <pre><code class='hljs language-markdown'>…highlighted source…</code></pre>
  </div>
  <div class='markdown-demo-parsed'>
    …rendered output…
  </div>
</div>

The container is laid out as a two-column CSS grid (grid-template-columns: 1fr 1fr), so the source and output sit side by side on a wide screen. The two label divs carry the headings Markdown code and Markdown output; the source lives in markdown-demo-markdown, the rendered result in markdown-demo-parsed. JMarkdown ships default styling for all of these classes, so demos look reasonable out of the box, and you can override any of it in your own stylesheet.

Class Holds
markdown-demo-container the whole grid
markdown-demo-code-label the Markdown code heading (left)
markdown-demo-output-label the Markdown output heading (right)
markdown-demo-markdown the highlighted source (left pane)
markdown-demo-parsed the rendered output (right pane)

How the two panes are produced

The left and right panes come from the same source text by two different routes, and the difference matters for a couple of edge cases below.

The right pane is parsed normally: JMarkdown runs the directive body through its full lexer and renderer, so every JMarkdown feature works inside a demo — inline directives, math, footnotes, custom elements, scripts, and even other container directives (with the colon-count caveat described under Nesting).

The left pane is not parsed as Markdown. The body is handed straight to highlight.js and displayed verbatim. This is deliberate: if the example you are demonstrating contains a fenced code block, or any other syntax that the lexer would try to consume, feeding it through the parser a second time just to colour it would corrupt it. By syntax-highlighting the raw text instead, the left pane can show anything — including fenced code, raw HTML, or a half-finished snippet — without risk.

One small consequence of the verbatim handling: the directive strips a single leading blank line from the source before highlighting, so the left pane is not pushed down by the newline that naturally follows the opening fence. You do not need to do anything for this; it just keeps the panel tidy.

Choosing the highlight language: {type=...}

By default the left pane is highlighted as Markdown, which is right for the overwhelming majority of demos. When the example you are showing is not Markdown — most often raw HTML or a script — pass a type attribute on the opening fence to tell highlight.js what language the source is in:

:::markdown-demo{type=html}
<button onclick="alert('hi')">Press me</button>
:::

Because that source contains raw HTML, the {type=html} variant highlights it as HTML (and, just as importantly, keeps highlight.js from misreading the angle brackets as Markdown). Here is the result:

Markdown output
<button onclick="alert('hi')">Press me</button>

The right pane is rendered the same way regardless of type — the attribute only controls how the source is coloured on the left. The value is passed straight to highlight.js as the language name, so any language highlight.js knows is valid: html, js, css, xml, latex, json, and so on. If you omit type, the language is markdown.

A reliable rule of thumb: if the demo's source contains a literal HTML tag or a script element, reach for {type=html}. A demo that embeds a jmarkdown script block (a script element whose data-type is jmarkdown) is the canonical case — the surrounding panels throughout Inline scripts and Block environments all use the {type=html} variant for exactly this reason.

Nesting demos around other directives

A markdown-demo is itself a container directive, delimited by colon fences. That creates one wrinkle you must know about: if the example you are demonstrating is itself a container directive, the demo's fence must use more colons than the inner one. This is the ordinary colon-counting rule for nesting container directives — the closing fence is matched by colon count, so an inner three-colon closer would otherwise close the demo prematurely and fold the rest of your document into the panel.

So to demonstrate a three-colon directive, wrap it in a four-colon demo (the example below shows a strategic-form game, a three-colon directive in its own right):

::::markdown-demo
:::game
         Rock   & Paper  & Scissors
Rock     & (0,0)  & (-1,1) & (1,-1)
Paper    & (1,-1) & (0,0)  & (-1,1)
Scissors & (-1,1) & (1,-1) & (0,0)

row: Player 1
column: Player 2
caption: Rock-Paper-Scissors
:::
::::

The four-colon opening fence wraps the three-colon game directive cleanly:

If the inner example itself contains nested containers, bump the demo's count further still — five colons wrapping a four-colon example, and so on. JMarkdown registers markdown-demo at every fence width from three up to eight colons, so you have plenty of room to nest. The general principle: the outermost fence always has the highest colon count.

The same rule applies in the other direction. To show how a markdown-demo fence itself looks — as the very first example on this page does — put the literal opening fence inside a fenced code block (triple backticks), not inside another demo. A code fence does not interpret directives, so the colons are shown literally with nothing to count.

Attributes

Attribute Default Effect

Behaviour at a glance

  • Left pane — the source, syntax-highlighted by highlight.js, never re-parsed as Markdown. Safe for fenced code, raw HTML, and partial snippets.
  • Right pane — the source parsed and rendered by JMarkdown's full pipeline. Every JMarkdown feature works inside it.
  • One source — both panes derive from the same text, so a demo and its output can never disagree.
  • HTML only — markdown-demo is a documentation aid for the web. It is registered on the HTML parser, so it has no effect in LaTeX output; author your print material without it.

Gotchas

  • Colon counting. Wrapping a container directive in a demo requires a higher colon count on the demo than on the inner directive (four around a three-colon directive, and so on). A missing or mismatched closing fence folds the rest of the document into the panel.
  • Closing fence alone on its line. As with any container directive, the closing colon fence must be the only thing on its line.
  • Use {type=html} for HTML and scripts. Without it, highlight.js highlights raw HTML or a script element as Markdown, which looks wrong and can mangle the display of angle brackets.
  • Showing the demo syntax itself. To document the markdown-demo fence literally, put it in a fenced code block, not inside another demo.
