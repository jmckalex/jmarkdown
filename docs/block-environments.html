Block environments (@begin … @end)

A block environment wraps a span of your document in a named container, in the spirit of a LaTeX environment:

@begin(name)
  …content…
@end(name)

Because the closing tag names what it closes, environments nest by name — you can wrap a block in another, or drop one into the middle of another, without the colon-counting that JMarkdown's colon-delimited container directives require. The @ sigil is borrowed from texinfo's @example … @end example and is otherwise unused, so it never collides with your prose.

The real point of this page is that you can define your own environments — either inline in a script block or in a reusable file — and a single definition produces both polished HTML and LaTeX. (Throughout, the live demos below show the source on the left and its rendered result on the right.) We start with the syntax, then spend most of our time on defining environments.

The syntax

An opener takes a name and two optional arguments, one bracket type per job:

@begin(name)[label]{key="value" …}
       ^^^^  ^^^^^  ^^^^^^^^^^^^^^^
       name  label  attributes

The closer is always the bare @end(name). Both the [label] and the {attributes} are handed to your environment's renderer (as ctx.text and ctx.attrs — see below), so you decide what they mean.

What an undefined name does

If you use a name you have not defined, JMarkdown still does something sensible so the environment is useful out of the box:

Markdown code
Markdown output
@begin(warning)
A plain name becomes a `<div>` whose class is the name.
@end(warning)

@begin(pull-quote)
A *hyphenated* name becomes a custom element instead,
because the HTML spec requires a hyphen in a valid
custom-element name.
@end(pull-quote)

A plain name becomes a <div> whose class is the name.

A hyphenated name becomes a custom element instead, because the HTML spec requires a hyphen in a valid custom-element name.

You can force either form with a sigil — @begin(.name) is always a <div>, @begin(<name>) is always an element — or set the document-wide policy with the Block elements metadata key (hyphenated (default) · all · none). In LaTeX, every environment renders as \begin{name}[label]…\end{name} regardless. You do not have to define the environment to get a compilable document: the assembled preamble automatically provides a guarded no-op definition for any undefined name (deferred with \AtBeginDocument, so a real \newenvironment{name} of yours — from the LaTeX preamble key, a package, or your document class — always wins, wherever it appears). Until you define it, the body typesets as plain paragraphs and a [label] shows literally — the print twin of an unstyled <div> waiting for its CSS.

One naming caveat for print: a name that is already a TeX command — box, outer, middle, frame, … — cannot work as a LaTeX environment at all (TeX runs the existing command when the environment opens), so dual-output documents should avoid such names.

JMarkdown invents no meaning for the name — that is your job, which brings us to:

Defining your own environments

A definition is a name plus a handler — an object that says how the environment renders. The one required key is html; everything else is optional:

key meaning
html (ctx) => string — the HTML renderer (required)
latex (ctx) => string — the LaTeX renderer (optional; if omitted, the html output is reused in LaTeX mode)
render (ctx) => string — a single renderer used for every output format, when HTML and LaTeX would be identical
mode how the body is read: 'markdown' (default), 'verbatim' (kept raw — for code or TeX), or 'custom'
tokenize (body, token) => … — only for mode: 'custom', for environments that parse their own body
numbered true to auto-number the environment and make it cross-referenceable — see Numbered environments below
counter the counter group to count in (default: the name; give two environments the same value to share a counter)
refname the :cref word, e.g. 'exercise' or ['exercise','exercises'] (default: the name)
title the printed/heading word, e.g. 'Exercise' (default: the capitalised refname)

The renderer receives a single context object, ctx:

ctx field what it holds
ctx.inner the body, already rendered (HTML or LaTeX, to match the output)
ctx.text the [label] string, or undefined
ctx.attrs the {attributes} as an object — ctx.attrs.kind, ctx.attrs.id, … (numbers are coerced)
ctx.rawText the body as raw, unrendered source
ctx.name the environment name
ctx.format 'html' or 'latex'
ctx.parser the active marked parser, if you need to render sub-content yourself

Defining an environment in a script block

The most direct way — and the most LaTeX-like — is to define an environment in a jmarkdown script block (one with data-type="jmarkdown") with defineEnvironment(name, handler). The definition runs while the document is being read, so define an environment before the first @begin that uses it (just as you would \newenvironment before the environment in LaTeX).

Here a callout environment uses both the [label] and a {kind} attribute:

Markdown code
Markdown output
<script data-type="jmarkdown">
defineEnvironment('callout', {
	html: (ctx) =>
		`<aside class="callout ${ctx.attrs?.kind ?? 'note'}">
			<strong>${ctx.text ?? ''}</strong> ${ctx.inner}
		</aside>`,
});
</script>

@begin(callout)[Tip]{kind="info"}
Define it once, then use it like */any/* other
environment — the body is processed as markdown, so
*emphasis* and $math$ work inside it.
@end(callout)
Tip

Define it once, then use it like any other environment — the body is processed as markdown, so emphasis and $math$ work inside it.

Because the same ctx is available whatever the output, you give the environment a LaTeX form simply by adding a latex key — one source, two outputs:

<script data-type="jmarkdown">
defineEnvironment('callout', {
	html:  (ctx) => `<aside class="callout ${ctx.attrs?.kind ?? 'note'}"><strong>${ctx.text ?? ''}</strong> ${ctx.inner}</aside>`,
	latex: (ctx) => `\\begin{callout}{${ctx.attrs?.kind ?? 'note'}}[${ctx.text ?? ''}]\n${ctx.inner}\n\\end{callout}\n\n`,
});
</script>

With that definition, @begin(callout)[Tip]{kind="info"} produces a <aside class="callout info">… for the web and \begin{callout}{info}[Tip]…\end{callout} for print, from the same source.

Loading environments from a file

When you have a set of environments you want across many documents, keep them in a .js file and pull them in from the metadata header with Load environments: — exactly parallel to Load directives and Load extensions. Each export is a handler:

// my-environments.js
export const warning = {
	html:  (ctx) => `<aside class="warning"><strong>${ctx.text ?? ''}</strong>${ctx.inner}</aside>`,
	latex: (ctx) => `\\begin{warning}[${ctx.text ?? ''}]\n${ctx.inner}\n\\end{warning}\n\n`,
};

export const theorem = {
	html:  (ctx) => `<div class="theorem">${ctx.inner}</div>`,
	latex: (ctx) => `\\begin{theorem}\n${ctx.inner}\n\\end{theorem}\n\n`,
};

Then, in the document's metadata header, either name the exports you want…

Load environments: warning, theorem from my-environments.js

…or, if the file's default export is a { name: handler } map, load them all at once:

Load environments: my-environments.js

Paths are resolved relative to the document, so the file travels with your project. (For environments you want available to every project, list the file under the Environments key in a global or project .jmarkdown/config.json, using an absolute path.)

Declaring LaTeX packages from a handler

If your latex renderer relies on a package, declare it the way JMarkdown's own renderers do — call requirePackage('name') inside the renderer (also available: addPreamble(line) for raw preamble lines such as a \newtcolorbox, and addLatePreamble(line) for lines that must follow hyperref/cleveref). The declaration is usage-driven: the full-document preamble gains the \usepackage only when the environment actually renders, and the calls are harmless in HTML output. 

defineEnvironment('shaded', {
	html:  (ctx) => `<div class="shaded">${ctx.inner}</div>`,
	latex: (ctx) => {
		requirePackage('tcolorbox');
		return `\\begin{tcolorbox}\n${ctx.inner}\n\\end{tcolorbox}\n\n`;
	},
});

Numbered environments

The built-in theorem, figure, and equation environments number themselves and answer to :ref / :cref. Your own environments can do the same — add numbered: true and JMarkdown takes over the counting and cross-referencing, in both editions, while you keep control of the box:

Markdown code
Markdown output
<script data-type="jmarkdown">
defineEnvironment('exercise', {
  numbered: true,
  html: (ctx) => `<aside class="exercise">${ctx.inner}</aside>`,
});
</script>

@begin(exercise)[Warm-up]{id=ex:warm}
Show that the square of an odd number is odd.
@end(exercise)

@begin(exercise)
And another, unlabelled.
@end(exercise)

The first is :cref[ex:warm].

Exercise 1 (Warm-up). Show that the square of an odd number is odd.

Exercise 2. And another, unlabelled.

The first is exercise 1.

JMarkdown prepends an "Exercise N." label (the optional [text] becomes the parenthetical note, as with theorems), records the {id=…} so :ref → "1", :cref → "exercise 1", :Cref → "Exercise 1", and — because the number is stamped on the environment — a :label placed inside the body resolves to it too. The :cref word defaults to the name; override it with refname (and title for the printed word) when the environment name and the reference word should differ.

Sharing a counter. Give two environments the same counter and they number in one interleaved sequence — an exercise followed by its solution becomes "Exercise 1", "Solution 2", "Exercise 3":

defineEnvironment('exercise', { numbered: true,
  html: (ctx) => `<aside class="exercise">${ctx.inner}</aside>` });
defineEnvironment('solution', { numbered: true, counter: 'exercise',
  html: (ctx) => `<aside class="solution">${ctx.inner}</aside>` });

What happens in print. With no latex renderer, a numbered environment is emitted as a thmtools theorem-like — the engine numbers it, and cleveref names it to match the web wording. (Shared counters use thmtools' sibling=.) Supply your own latex renderer and you take over the LaTeX side entirely, including its counter. Either way the HTML side is unchanged.

Note

For a numbered environment, let JMarkdown own the id — put it in {id=…} on the @begin, not inside your html. (Sharing a counter with a built-in theorem is not supported yet; share among your own environments.)

Built-in environments

A handful of names are predefined so that @begin(x) matches its corresponding container directive exactly:

  • abstract and feedback — styled containers for the web;
  • TeX — raw LaTeX, passed through verbatim and emitted only in LaTeX output;
  • HTML — markdown prose emitted only in HTML output;
  • comment (and any names you declare under Optionals) — hidden by default, shown with @begin(comment){include=true};
  • game — a strategic-form game (see Strategic-Form Games).

You can use these immediately, and you can define your own alongside them with either method above.
