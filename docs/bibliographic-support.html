Bibliographic support (\cite{…})

A scholarly document needs citations and a bibliography. JMarkdown gives you both with a deliberately familiar interface: you write LaTeX-style citation commands — \cite{…}, \citep{…}, \nocite{…}, and friends — directly in your prose, point JMarkdown at a BibTeX .bib database and a citation style, and a fully formatted bibliography is assembled for you.

There are three ways that source can turn into finished citations, and the one you get depends on the output format and one metadata key:

  • Compile-time HTML (recommended). With Resolve citations: true, the bibliography is assembled while the document is processed. The inline citations and the reference list are baked into the HTML; the file is finished the moment it is written. This is the right default for a static web page or a printable web document — open it offline and everything is already resolved.
  • Runtime HTML (the alternative). Without that key, JMarkdown leaves your \cite{…} commands literal and wires the page up to the browser Biblify client, which resolves them when the page loads. Choose this when the document's content is revealed dynamically — the original reveal.js slide-deck use case — so there is nothing to resolve until the browser builds it.
  • LaTeX (native natbib). With --to latex, the citation commands pass through verbatim to natbib and ::Bibliography emits \bibliographystyle and \bibliography. Real bibtex/biber does the work when you compile the .tex.

The same \cite-family grammar, the same Bibliography database, and the same style name drive all three, so one source feeds a web edition and a print edition without rewriting a single citation.

Showing the syntax versus using it. Every citation command on this page is shown inside a code span or a fenced code block. The compile-time tokenizer never fires inside code (and the runtime client is not active here), so these examples display as literal syntax rather than resolving. When you write a real \cite{…} in running prose, type it plainly and it is processed normally.

Compile-time HTML resolution (recommended)

Switch compile-time resolution on with three keys in the document's metadata header (or in a configuration file): 

Title: A Paper About Conventions
Resolve citations: true
Bibliography: references.bib
Bibliography style: chicago


# Introduction

As argued in \cite{Lewis:1969}, …

::Bibliography
Metadata key Purpose
Resolve citations true to assemble citations and the bibliography at build time. Absent (the default) leaves the runtime path in charge.
Bibliography Path to a BibTeX .bib database, resolved relative to the source file.
Bibliography style A built-in style name or a path to a CSL file (see below).

When Resolve citations is on, processing does the following, with no browser involved:

  1. JMarkdown reads and indexes your .bib file from disk.
  2. A post-pass formats every citation with citation.js and the Citation Style Language (CSL), replacing each \cite{…} with a finished inline citation.
  3. The ::Bibliography directive is replaced by the assembled reference list, in citation.js's standard div.csl-bib-body structure.
  4. The runtime client scripts are suppressed — there is nothing left for the browser to do, so the page carries no Biblify JavaScript at all.

The engine is citation-js, the same library the runtime client uses, so compiled output matches what the browser would have produced. Because everything is resolved in the file, a compile-time document opens correctly offline and does not depend on the .bib being reachable at view time.

Tooltips and a trimmed .bib

Two further keys refine the compile-time output:

Metadata key Effect
Citation tooltips true to attach a hover tooltip to each inline citation showing the full reference.
Minimal bibliography true to write a trimmed <output>.cited.bib next to your HTML, containing only the entries actually cited.

Minimal bibliography needs a real output file to sit beside, so it is skipped (with a console note) when output goes to stdout. The trimmed database is handy for shipping a paper with exactly the references it uses and nothing more.

The citation commands

The \cite-family grammar is shared by all three paths. Every command names one or more keys from your .bib file inside braces; multiple keys are separated by commas.

Command Result
\cite{key} Textual (author-as-subject) citation — author names followed by a parenthesised year, e.g. Lewis (1969).
\citet{key} The same as \cite — t for textual; the default when no suffix is given.
\citep{key} Parenthetical citation — everything in parentheses, e.g. (Lewis 1969).
\citeauthor{key} The author names only, with no year.
\citeauthor*{key} The author names with full given names, and all authors listed (no et al. truncation).
\nocite{key} Adds the entry to the bibliography but prints nothing inline. Accepts several comma-separated keys at once.
\fullcite{key} Inserts the entry's complete bibliography-style reference inline, at that spot.

Two optional arguments in square brackets, before the brace, work exactly as in natbib: a lone [postnote] is appended inside the citation, and a [prenote][postnote] pair brackets it on both sides. They are most useful with the parenthetical form:

Source Renders as
\citep{Lewis:1969} (Lewis 1969)
\citep[42]{Lewis:1969} (Lewis 1969, 42)
\citep[see][ch. 3]{Lewis:1969} (see Lewis 1969, ch. 3)
\cite[42]{Lewis:1969} Lewis (1969, 42)

A few details worth knowing:

  • Multiple keys. \citep{Axelrod/Hamilton:1981,Skyrms:1996} produces a single parenthetical group listing both works, separated by a semicolon.
  • The default is textual. \cite{…} behaves like \citet{…}; reach for \citep{…} when you want the author and year fully inside parentheses.
  • \nocite pads a bibliography. Use it for works you want listed but do not cite in the body — one key at a time or several at once.
  • Keys with _ or / are safe. The tokenizer claims each whole command before any other inline rule runs, so a key like Axelrod/Hamilton:1981 is never mangled by the /italics/ or subscript syntax, and a literal example in a code span is left alone.

Citation styles

Bibliography style accepts either a recognised style name or the path to your own CSL file. The names handled directly are:

Name Style
apa American Psychological Association (author–date)
harvard1 Harvard, numbered variant
vancouver Vancouver (numeric, whole-document)
chicago Chicago Manual of Style (author–date)
bjps British Journal for the Philosophy of Science
ajp Australasian Journal of Philosophy
econometrica Econometrica
ergo Ergo

The first four (apa, harvard1, vancouver, chicago) plus bjps are recognised as built-in names directly. The journal styles ajp, econometrica, and ergo ship as bundled CSL files in JMarkdown, so naming them works too — they resolve to the bundled template of the same name.

Any other value is treated as a path to a CSL file (a .csl or .xml document):

Bibliography style: my-journal.csl

JMarkdown registers the file under a name derived from its basename and formats your citations and bibliography with it. This is how you support a journal whose style is not bundled: download its CSL file (the Zotero Style Repository is a comprehensive source), drop it beside your document, and name it here. The path resolves relative to the source file.

Runtime browser Biblify (the alternative)

The original mechanism resolves citations in the browser instead of at build time. You enable it with Biblify activate: true and without Resolve citations:

Title: A Set of Slides
Biblify activate: true
Bibliography: references.bib
Bibliography style: chicago


# First slide

As argued in \cite{Lewis:1969}, …

JMarkdown then leaves every \cite{…} command literal in the HTML and loads the Biblify client (citation.min.js + biblify.js). When the page loads, the client fetches your .bib over a relative URL, scans the rendered document, replaces each command with a formatted inline citation, and appends the bibliography to the end of the page (after any footnotes).

By default the runtime client also adds, for convenience, a Bibliography heading (an h1 with id="bib") and — if the document has a table of contents — a Bibliography entry linking to it. Each inline citation is wrapped in a span carrying a data-bibtex attribute recording its key(s), so citations are easy to style.

Reach for this path when resolution must happen in the browser — most often because the cited content is built or revealed dynamically. For an ordinary static document, prefer compile-time resolution: it needs no JavaScript at view time and the .bib need not be reachable from where the page is served.

Fine-tuning the runtime client

Beyond the activation keys, the runtime client's behaviour is controlled through the nested Biblify object in a configuration file:

Config key (under Biblify) Default Effect
add helper function true Inject the script that adds the heading and TOC entry.
add section heading true Place a Bibliography heading above the list.
add toc entry true Add a Bibliography link to the table of contents.
bibliography "" The .bib database path (set via the Bibliography metadata key).
bibliography style "" The style name or CSL path (set via Bibliography style).
defer false If true, the client configures itself but does not auto-process on load.

Deferred processing

Setting Bibliography defer: true in the metadata header (or "defer": true in the config) tells the client to load and configure itself but not to scan and replace citations automatically on load. This is for documents whose content appears over time — the reveal.js slides case — so you trigger processing yourself once the relevant content exists. Deferring also suppresses the heading/TOC helper, since there may be nothing to process yet. For an ordinary static document, leave defer at its default of false.

LaTeX output (native natbib)

With --to latex, JMarkdown does not format anything itself — it hands the work to LaTeX:

  • The \cite-family commands pass through verbatim as native natbib commands. The one exception is \fullcite, which natbib lacks: it is translated to \bibentry (from the bibentry package).
  • The ::Bibliography directive emits \bibliographystyle{…} followed by \bibliography{…}, pointing at the basename of your Bibliography database.
  • Real bibtex/biber resolves the citations and builds the reference list when you compile the resulting .tex.

Two things are therefore on you in the surrounding document. JMarkdown's --to latex output is body-only, so add \usepackage{natbib} (and \usepackage{bibentry} if you use \fullcite) in your own preamble. The .bst style file used by \bibliographystyle is chosen from Bibliography style by default; override it explicitly with the LaTeX bib style key:

Metadata key Effect
LaTeX bib style The natbib .bst name for \bibliographystyle in LaTeX output (e.g. apalike, plainnat). Overrides the default derived from Bibliography style.

Gotchas and limitations

  • Pick a path with one key. Resolve citations: true gives you compile-time HTML and suppresses the runtime scripts; omit it (with Biblify activate: true) for browser-side resolution. The two are mutually exclusive for a given document.
  • Resolve citations needs a Bibliography. If it is on but no database is given, JMarkdown logs an error and leaves the citation commands literal in the output rather than emitting empty placeholders.
  • The .bib path resolves differently per path. Compile-time reads the file from disk relative to the source file. The runtime client fetches it over a relative URL resolved against the output HTML file, so for the runtime path the .bib must be served from somewhere the browser can reach.
  • One database, one default style per document. A single .bib and one document-wide style apply; in compile-time HTML a per-::Bibliography style override and scope-based sectional lists are available, but the LaTeX path emits one \bibliography for the whole document.
  • Vancouver is whole-document. It produces a single numbered reference list and collapsed numeric in-text citations; per-section partitioning and per-list style switching do not apply to it.
  • A missing key is reported, not fatal. A \cite to a key absent from the .bib logs a warning and is skipped; the build still completes.
