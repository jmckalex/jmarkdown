Introduction

JMarkdown is a JavaScript-empowered variant of Markdown with dynamic, in-document syntax extensions.

JMarkdown is a Markdown authoring system built on marked.js. You write a single source file and JMarkdown turns it into polished HTML for the web — or, increasingly, into LaTeX for print. It began as a way to satisfy one author's preferred Markdown syntax and grew into a full-featured interpreter whose defining trait is this: you can invent new syntax inside the Markdown file itself, and you can run JavaScript while the document is being processed to generate or transform the output.

This page is the guided tour. It explains what makes JMarkdown different from ordinary Markdown, shows the headline features in action, and points you to the reference pages where each feature is documented in full. If you want the why rather than the what, read Motivation first; if you want the complete list of header keys, see the Metadata Header.

Throughout this manual, examples are shown with a live demo — the source on the left and its actual rendered output on the right, generated by JMarkdown as the page is built. Because the demo and the result come from the same source, an example can never silently drift out of date.

What JMarkdown gives you

A plain Markdown processor has a fixed vocabulary: emphasis, headings, lists, links, code. JMarkdown keeps all of that and adds four capabilities that, between them, account for most of what the rest of this manual documents.

  1. A nicer base syntax. JMarkdown adjusts Markdown's inline notation and layers on conveniences — better emphasis markers, real subscripts and superscripts, highlight and underline, emoji and Font Awesome icons, alignment, and more. See Syntax adjustments.
  2. Author-defined syntax. You can declare brand-new inline notation in the document's metadata header, with no JavaScript at all — either as a fixed pair of delimiters (simple form) or as a regular-expression pattern (complex form).
  3. Embedded JavaScript. A script block whose data-type is jmarkdown runs at processing time. It can define functions you call from the prose, generate HTML directly, or post-process the finished document. See Inline scripts.
  4. Block-level structure. Container directives and named block environments give you LaTeX-style environments — and a single definition can emit both HTML and LaTeX.

The sections below take the distinctive three of these in turn: author-defined inline syntax, embedded JavaScript, and the live-demo convention that ties the manual together.

Inventing your own inline syntax

The feature that gives JMarkdown its character is that new inline notation can be defined in the Markdown file itself — declared once in the metadata header and then available everywhere in that document. No plugin to install, nothing to register by hand.

Extensions come in two flavours. A simple extension names a fixed opening delimiter, a fixed closing delimiter, and a chunk of replacement HTML. A complex extension trades the literal delimiters for two regular expressions, so it can recognise a pattern and capture pieces of it. Both share the same replacement machinery: the captured text drops into the template at the ${content1}, ${content2}, … slots.

Unicode delimiters

Because JMarkdown is fully Unicode-aware, you are not confined to the handful of ASCII punctuation marks that base Markdown already claims. You can choose delimiters that look like what they do. The metadata header of this very page defines two such extensions — arrows for small caps, corner brackets for a TeXbook-style string: 

Extension smallcaps: ↑ ↑ false 1
	<span style="font-variant-caps: small-caps;">${content1}</span>
Extension texbook: ⌜ ⌝ false 1
	<span class="texbook" style="font-family: 'Fira Code';">${content1}</span>

With those in place, the notation reads naturally and renders live on this page:

Markdown code
Markdown output
Press ↑Save↑ to keep your work, and consult ⌜The TeXbook⌝ for the typography.

Press Save to keep your work, and consult The TeXbook for the typography.

Each definition is two parts: a header line — naming the extension, giving its opening delimiter, its closing delimiter, whether the body is re-parsed as Markdown (false here), and how many arguments it captures (1) — followed by the indented replacement HTML in which ${content1} stands for whatever the author typed between the delimiters. The full rules are on the simple-extension page.

Pattern-matching with regular expressions

Fixed delimiters cannot recognise a shape. When you want to match patterned prose — variables written x1, y2, z3; grades in brackets; a labelled identifier — you give the extension two regular expressions instead. On the old mathematics newsgroups it was conventional to write subscripted variables as p1, p2, pk; this page defines a complex extension that recognises exactly that shape and typesets it with MathJax:

Extension subscripts:  /[pxyz][0-9k]+/	/^([pxyz])([0-9k]+)/	false 2
	<span>\(${content1}_{${content2}}\)</span>

The result is automatic, anywhere the pattern appears in your prose:

Markdown code
Markdown output
Consider points p1, p2, p3, … where pk is the point (xk, yk, zk).

Consider points \(p_{1}\), \(p_{2}\), \(p_{3}\), … where \(p_{k}\) is the point \((x_k, y_k, z_k)\).

The first regular expression is a cheap scan that locates a candidate; the second is the real matcher, whose parenthesised capture groups become ${content1} and ${content2}. A second extension on this page captures the three components of the tuple (xk, yk, zk) and renders them as a properly subscripted triple. The complex-extension page covers capture groups, the parse modes, and the gotchas in detail.

Running JavaScript at processing time

The second pillar is that JMarkdown can evaluate JavaScript while it processes the document. You place the code in a script block whose data-type is jmarkdown; the code runs in a Node virtual machine whose context is shared across every such block, so a function defined in one block is available in all of them. The block itself is removed from the output. With export_to_jmarkdown("name") you make a function callable from the prose, where name(...) is replaced by whatever the function returns.

This page defines a rainbow function that wraps each character of a string in an animated span. It is a deliberately playful example, but it shows the shape of the mechanism: a function defined in JavaScript, exported to the document, and then invoked inline as if it were ordinary notation.

A bare call inserts the function's return value where the call appears, and you can pass an options object as a second argument:

Markdown code
Markdown output
rainbow("It's time to be awesome!")

rainbow("And the colours can vary!",
    {duration: 3, delay: 0.08,
     colors: ['#000000', '#666666', '#ffffff',
              '#666666', '#000000']})

It's time to be awesome!

And the colours can vary!

JavaScript in JMarkdown is not limited to text generation. The same mechanism lets you generate HTML programmatically, and a related post-processing pass — built on Cheerio for light DOM manipulation — lets you rewrite the finished HTML after rendering. The Inline scripts, Post-processing, and detailed scripting example pages develop all of this.

Live demos: examples that cannot drift

You will have noticed the side-by-side panels above. Each is produced by the markdown-demo container directive, which takes a block of source, shows it on the left exactly as written, and shows JMarkdown's rendered output of that same source on the right: 

:::markdown-demo
:span[A coloured span.]{.goldenrod}

1. A list *containing Markdown*
2. Isn't that /cool/?
:::

Because the demonstrated source and the displayed result are one and the same, a demo is guaranteed to stay in sync with the feature it documents. When an example contains raw HTML or a script that is meant to run — as the rainbow demo above does — add the {type=html} attribute so the left panel is highlighted as HTML. Full details are on the Demonstrating Markdown page.
